# 架构文档

## 文档概述

本文档描述道一（UniDo）系统的分层架构、架构原则、关键层职责以及层内各组件职责。

## 架构原则

### 1. 关注点分离（Separation of Concerns）

每层只关注特定的职责：
- **L4 Gateway**：只关注外部通信协议（HTTP、WebSocket）
- **L3 Scenario**：只关注业务场景的协调和编排
- **L2 Domain**：只关注领域逻辑和业务规则
- **L1 Infrastructure**：只关注技术基础设施

### 2. 依赖倒置（Dependency Inversion）

高层模块不依赖低层模块，两者都依赖抽象：
- L3 调用 L2 的接口，而非具体实现
- L2 调用 L1 的接口，而非具体实现
- 通过依赖注入实现解耦

### 3. 单一职责（Single Responsibility）

每个组件只负责一个明确的功能：
- `MessageService` 只负责消息的 CRUD
- `LLMClient` 只负责 LLM API 调用
- `EventBus` 只负责事件发布和订阅

### 4. 开放封闭（Open/Closed）

系统对扩展开放，对修改封闭：
- 新增工具只需添加描述文件和实现
- 新增 LLM 适配器只需实现 `BaseAdapter` 接口
- 新增事件类型只需在 `EventTypes` 中定义

### 5. 事件驱动（Event-Driven）

系统通过事件总线进行异步通信：
- 组件间通过事件解耦
- 支持事件的持久化和回放
- 支持事件的订阅和过滤

## 各层职责详解

### L1 Infrastructure Layer - 基础设施层

**层职责**：提供技术基础设施，为上层提供通用能力。

#### L1a ID Generator - ID生成器

| 组件 | 职责 | 关键方法 |
|------|------|----------|
| `id_generator.py` | 生成各类唯一ID | `generate_request_id()`、`generate_message_id()`、`generate_call_id()` |

**ID格式规范**：
```
request_id: req-{uuid8}      例: req-a1b2c3d4
message_id: msg-{uuid8}      例: msg-e5f6g7h8
call_id: call-{uuid8}        例: call-i9j0k1l2
session_id: sess-{uuid8}     例: sess-m3n4o5p6
dialog_id: dialog-{uuid8}    例: dialog-q7r8s9t0
```

#### L1b Persistence - 持久化服务

| 组件 | 职责 | 关键方法 |
|------|------|----------|
| `storage_factory.py` | 创建存储实例 | `create()` |
| `file_storage.py` | JSON文件存储 | `save()`、`load()`、`list()`、`delete()` |
| `persistence_service.py` | 持久化服务门面 | `save()`、`get()`、`list()` |

**存储类型**：
```
storage_types:
  - 'sessions'     → 会话数据
  - 'dialogs'      → 对话数据
  - 'messages'     → 消息数据
  - 'models'       → 模型配置
  - 'workspaces'   → 工作区配置
  - 'projects'     → 项目配置
  - 'tools'        → 工具定义
  - 'prompts'      → 提示词模板
  - 'events'       → 事件记录
  - 'llm_calls'    → LLM调用记录
```

#### L1c LLM Client - LLM客户端

| 组件 | 职责 | 关键方法 |
|------|------|----------|
| `llm_client.py` | LLM API调用门面 | `call()`、`call_stream()`、`set_mode()` |
| `api_adapters/` | API适配器集合 | `OpenAIAdapter`、`AnthropicAdapter`、`QwenAdapter` |
| `response_parser.py` | 响应解析器 | `parse_stream_chunk()`、`parse_response()` |

**适配器接口**：
```python
class BaseAdapter:
    def prepare_request(self, request: LLMRequest) -> Dict
    def parse_response(self, response: Dict) -> LLMResponse
    def parse_stream_chunk(self, chunk: Dict) -> StreamChunk
```

#### L1d Events - 事件总线

| 组件 | 职责 | 关键方法 |
|------|------|----------|
| `event_bus.py` | 事件发布/订阅 | `publish()`、`subscribe()`、`unsubscribe()` |
| `event_types.py` | 事件类型定义 | `EventTypes` 常量类 |
| `event_record.py` | 事件记录模型 | `Event`、`EventRecord` |
| `subscription.py` | 订阅管理 | `SubscriptionManager` |

**事件类型分类**：
```
系统事件:
  - system.initialized
  - system.shutdown

会话事件:
  - session.created
  - session.switched

对话事件:
  - dialog.created
  - dialog.round_started

消息事件:
  - message.created

LLM事件:
  - llm.request_sent
  - llm.call_thinking
  - llm.call_reasoning
  - llm.call_text
  - llm.call_completed

工具事件:
  - tool.call_started
  - tool.execution_output
  - tool.call_completed

配置事件:
  - model.created
  - workspace.created
  - project.created
```

#### L1e Storage Config - 存储配置

| 组件 | 职责 | 关键方法 |
|------|------|----------|
| `storage_config_service.py` | 存储策略配置 | `should_persist_event()` |
| `event_storage_config_service.py` | 事件持久化配置 | `get_config()`、`update_config()` |

### L3 Scenario Coordination Layer - 场景协同层

**层职责**：协调多个领域服务，编排业务场景。

#### L3a Task Coordination - 任务协调

| 组件 | 职责 | 关键方法 |
|------|------|----------|
| `dialogue_service.py` | 对话服务 | `process_dialogue()`、`init_dialog()` |
| `base_execution_service.py` | 基础执行服务 | `execute_with_recursive_llm()` |
| `dialogue_based_llm_service.py` | 对话式LLM服务 | `call_llm()` |
| `intent_service.py` | 意图分析服务 | `analyze_intent()` |
| `tool_task_executor.py` | 工具任务执行器 | `execute()` |
| `task_group_executor.py` | 任务组执行器 | `execute()` |

**核心协调流程**：
```
DialogueService.process_dialogue():
  1. 创建对话 (DialogService)
  2. 创建用户消息 (MessageService)
  3. 发布消息事件 (EventBus)
  4. 执行LLM循环 (BaseExecutionService)
     ├─ 调用LLM (DialogueBasedLLMService)
     ├─ 解析工具调用
     ├─ 执行工具 (ToolExecutor)
     ├─ 创建工具消息 (MessageService)
     └─ 再次调用LLM
  5. 返回最终响应
```

## 架构设计决策

### 1. 为什么采用四层架构？

| 问题 | 解决方案 |
|------|----------|
| 协议变化（HTTP→gRPC） | 只修改L4 Gateway |
| 业务场景变化 | 只修改L3 Scenario |
| 领域规则变化 | 只修改L2 Domain |
| 技术基础设施变化 | 只修改L1 Infrastructure |

### 2. 为什么采用事件驱动？

| 问题 | 解决方案 |
|------|----------|
| 组件间耦合 | 通过事件解耦，发布者不关心订阅者 |
| 异步处理 | 事件异步发布，不阻塞主流程 |
| 状态追踪 | 事件持久化，支持回放和审计 |
| 前后端同步 | WebSocket推送事件，实时更新UI |

### 3. 为什么采用JSON文件存储？

| 问题 | 解决方案 |
|------|----------|
| 快速开发 | 无需数据库配置，直接文件读写 |
| 易于调试 | JSON格式可直接查看和修改 |
| 可扩展性 | 后期可替换为数据库，接口不变 |
