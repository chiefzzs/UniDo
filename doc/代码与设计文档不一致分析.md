# 代码与设计文档不一致分析报告

## 概述

本报告详细分析了 `src` 目录下现有代码与设计文档之间的不一致之处，包括数据结构、控制流、存储格式等方面的差异，并针对每个不一致项给出设计文档更新建议。

---

## 架构设计原则

### 核心原则：L1b 不应定义业务实体的具体数据结构

**问题背景：**
在不一致分析过程中，发现设计文档中存在 L1b 持久化服务定义具体业务实体（Project、Session、Message、LLMCallRecord 等）数据结构的情况。这种设计会导致：

- **职责耦合**：L1b（基础设施层）不应该关心业务实体的具体结构
- **违反分层架构**：业务实体定义属于领域层（L2）的职责，不应放在基础设施层
- **维护成本增加**：当业务实体变更时，需要修改 L1b

**架构原则定义：**

| 层级 | 职责 | 不应包含 |
|-----|------|---------|
| **L1b 持久化基础设施** | 提供通用的 JSON 文件读写能力、环境隔离、文件系统操作 | 业务实体的具体数据结构定义 |
| **L2 层领域服务** | 定义业务实体结构、数据验证、业务规则 | 文件系统操作细节 |
| **调用方式** | L2 将实体序列化为 dict → 调用 L1b.save() → L1b 写入文件 | 绕过 L1b 直接操作文件 |

**L1b 核心接口（通用、无业务耦合）：**

| 方法 | 功能 | 参数 | 返回值 |
|-----|------|------|-------|
| `save(entity_type, data)` | 保存实体 | `entity_type: str`, `data: dict` | `entity_id: str` |
| `load(entity_type, entity_id)` | 加载实体 | `entity_type: str`, `entity_id: str` | `Optional[dict]` |
| `list(entity_type, filters=None)` | 列出实体 | `entity_type: str`, `filters: dict` | `List[dict]` |
| `delete(entity_type, entity_id)` | 删除实体 | `entity_type: str`, `entity_id: str` | `bool` |

**存储结构（由 L1b 定义，业务层使用）：**

```
data/{env}/
├── projects.json      # L2 定义格式，L1b 只负责读写
├── sessions.json      # L2 定义格式，L1b 只负责读写
├── llm_calls.json     # L2 定义格式，L1b 只负责读写
├── events.json        # L1d 定义格式
├── prompts.json       # L2 定义格式，L1b 只负责读写
└── model_configs.json # L2 定义格式，L1b 只负责读写
```

**好处：**

- **解耦**：L1b 可以独立演进，不依赖业务变化
- **复用**：L1b 可以服务于不同的业务模块
- **清晰**：职责边界明确，符合分层架构原则

**影响的不一致项：**

- 1.1 持久化存储格式不一致 - 移除 L1b 中的数据类定义
- 2.3 Message 实体定义重复 - 改为由 L2 层统一定义

## L1 基础设施层

### 1. 持久化存储格式不一致

#### 设计文档要求（L1b_持久化与IO基础设施.md）
- 每种类型的信息使用一个独立的JSON文件
- 多个数据采用数组格式存储
- 文件命名规范：`{entity_type}.json`（如 `projects.json`、`sessions.json`）
- 支持环境隔离（`STORAGE_ENV=dev/test`）
- 存储路径：`data/{env}/{entity_type}.json`

#### 实际代码实现（S01_data_persistence.py）
- 每个实体使用单独的文件：`{entity_id}.json`
- 实体存储在按类型分组的目录中：`data/{env}/{entity_type}/{entity_id}.json`
- 配置文件支持单文件模式和多配置模式

**不一致说明：**
- 设计要求单文件数组格式，代码实现为多文件目录结构
- 设计要求所有实体使用统一格式，代码中配置文件和实体文件格式不同

**影响：**
- 不符合设计文档的"完美JSON"要求，人工核查困难
- 无法直接查看所有项目/会话的完整列表
- 增加了文件管理的复杂度

**设计文档更新建议：**
1. **确认存储格式**：明确采用"单文件数组格式"，每个实体类型对应一个 JSON 文件
2. **统一文件结构**：
   ```
   data/{env}/
   ├── projects.json       # 所有项目的数组
   ├── sessions.json       # 所有会话的数组（内嵌 messages）
   ├── llm_calls.json      # 所有LLM调用记录的数组
   ├── events.json         # 所有事件的数组
   ├── prompts.json        # 所有提示词的数组
   ├── workspace_configs.json
   └── model_configs.json
   ```
   - **说明**：Message 内嵌于 Session，不独立存储（符合 DDD 聚合根原则）
3. **通用存储接口**：L1b 提供通用的 CRUD 接口（save/load/list/delete），不定义具体业务数据结构
4. **业务实体由 L2 定义**：Project、Session、LLMCallRecord 等业务实体的数据结构由 L2 层定义，L1b 只负责存储
5. **明确迁移策略**：提供从多文件目录结构到单文件数组格式的数据迁移方案

### 2. 环境变量名称不一致

#### 设计文档要求
- 环境变量：`STORAGE_ENV`
- 值：`dev` 或 `test`

#### 实际代码实现（S01_data_persistence.py, S21_llm_invocation.py）
- 环境变量：`AI_ASSISTANT_ENV`
- 值：`dev` 或 `test`

**不一致说明：**
- 环境变量名称不同

**影响：**
- 用户需要使用不同的环境变量名，造成混淆
- 文档与代码不匹配

**设计文档更新建议：**
1. **统一环境变量名称**：统一使用 `STORAGE_ENV` 作为环境隔离变量
2. **明确变量定义**：在 L1_基础设施层总览.md 的 7.4.1 节明确：
   ```
   STORAGE_ENV: 环境变量，用于隔离开发与测试环境
   - 值: "dev" | "test"
   - 默认值: "dev"
   - 影响范围: 所有 L1 层组件的数据存储路径
   ```
3. **更新代码实现**：将 `AI_ASSISTANT_ENV` 统一替换为 `STORAGE_ENV`

### 3. 事件持久化方式不一致

#### 设计文档要求（L1d_事件管理基础设施.md）
- 事件通过 L1b 持久化到 `events.json` 文件
- 事件以数组格式存储
- 支持序列查询和回放
- 事件发布后通过 L1b 的 `save_events()` 方法存储

#### 实际代码实现
- S03_event_plugin.py：事件仅存储在内存中（`_event_history` 列表）
- S05_event_storage_plugin.py：存储路径为 `config/events.json`，不符合设计要求

**不一致说明：**
- 设计要求事件通过 L1b 持久化，代码中 S03_event_plugin.py 不依赖 L1b
- S05_event_storage_plugin.py 存储路径不符合设计要求

**影响：**
- 事件持久化逻辑分散在多个组件中
- 事件存储路径不符合环境隔离要求

**设计文档更新建议：**
1. **明确持久化职责**：在 L1d_事件管理基础设施.md 中明确：
   - L1d 依赖 L1b 进行事件持久化
   - 事件存储路径：`data/{env}/events.json`
   - 事件存储格式：数组格式，每项包含完整的事件信息
2. **更新 L1d 数据流**：
   ```
   事件发布 → L1d 事件系统 → 内存缓存 → L1b 持久化 → data/{env}/events.json
   ```
3. **新增方法**：在 L1b 中增加 `save_events(events: List[EventRecord])` 和 `get_events(criteria)` 方法
4. **删除独立存储**：移除 S05_event_storage_plugin.py 的独立存储逻辑，由 L1d 直接调用 L1b

### 4. LLM调用记录持久化缺失

#### 设计文档要求（L1c_大模型基础设施.md）
- LLM调用过程通过 L1b 持久化到 `llm_calls.json` 文件
- 调用记录以数组格式存储
- 支持序列存储

#### 实际代码实现（S21_llm_invocation.py）
- 支持缓存功能（`llm_cache` 目录）
- 支持录制回放功能（`recording` 目录）
- **没有专门的 LLM 调用记录持久化到 `llm_calls.json`**

**不一致说明：**
- 设计要求 LLM 调用记录持久化到 `llm_calls.json`，代码中未实现
- 缓存和录制功能不等同于调用记录持久化

**影响：**
- 无法追踪所有 LLM 调用历史
- 不符合设计文档的审计要求

**设计文档更新建议：**
1. **新增数据类**：在 L1b_持久化与IO基础设施.md 中新增 `LLMCallRecord` 数据类，以 `request` 和 `response` 对象为核心：
   ```json
   {
     "call_id": "llm-123",
     "session_id": "sess-123",
     "model_config_id": "model-xyz",
     "request": {
       "messages": [{"role": "user", "content": "用户问题"}],
       "model": "qwen-7b",
       "temperature": 0.7,
       "max_tokens": 4096,
       "stream": false
     },
     "response": {
       "choices": [{"message": {"role": "assistant", "content": "LLM响应"}}],
       "usage": {"prompt_tokens": 10, "completion_tokens": 20, "total_tokens": 30}
     },
     "status": "completed|failed|in_progress",
     "duration_ms": 1500,
     "created_at": "2026-05-18T10:00:00"
   }
   ```
2. **更新 L1c 文档**：
   - 明确 LLM 调用记录通过 L1b 持久化
   - 调用开始时创建记录，调用完成后更新记录状态
   - 支持按 session_id、time_range 等条件查询调用记录
3. **新增持久化方法**：在 L1b 中增加 `save_llm_call_record()` 和 `get_llm_call_records()` 方法

### 5. 配置数据结构不一致

#### 设计文档要求（L1a_统一配置管理.md）
- 配置通过 L1b 持久化到数组格式的 JSON 文件
- ModelConfig 字段：`entity_id`, `name`, `model_name`, `api_type`, `api_address`, `api_key`, `temperature`, `max_tokens`, `system_prompt`, `is_active`, `created_at`, `updated_at`

#### 实际代码实现（S02_config.py）
- ModelConfig 字段：`id`, `name`, `model_address`, `api_key`, `api_type`, `temperature`, `max_tokens`, `system_prompt`, `is_active`, `created_at`, `updated_at`, `mode`, `provider`, `stream`
- 支持字段映射：`model_name` ↔ `name`, `api_base` ↔ `model_address`

**不一致说明：**
- 字段名称不一致（`entity_id` vs `id`，`api_address` vs `model_address`）
- 代码中包含设计文档未定义的字段（`mode`, `provider`, `stream`）
- 支持字段映射，增加了复杂度

**影响：**
- 配置数据结构不统一
- 增加了字段映射的维护成本

**设计文档更新建议：**
1. **统一字段命名**：在 L1a_统一配置管理.md 中统一 ModelConfig 字段（移除 `system_prompt`，由 L2 层提示词管理模块管理）：
   ```
   字段名            | 类型      | 说明
   -----------------|----------|----
   config_id         | str      | 配置唯一标识（替代 entity_id）
   name              | str      | 配置名称（前端显示用）
   model_name        | str      | 模型名称（LLM 模型标识）
   api_type          | str      | API 类型 (qwen/openai/claude)
   api_address       | str      | API 地址（替代 model_address）
   api_key           | str      | API 密钥
   temperature       | float    | 温度参数
   max_tokens        | int      | 最大 token 数
   is_active         | bool     | 是否激活
   stream            | bool     | 是否流式输出（新增）
   prompt_id         | str      | 关联的提示词ID（引用 L2 层提示词管理模块）
   created_at        | datetime | 创建时间
   updated_at        | datetime | 更新时间
   ```
2. **移除冗余字段**：删除 `mode`, `provider`, `system_prompt` 等冗余字段
3. **新增提示词管理模块**：在 L2 领域层新增提示词管理模块（PromptManagementService），负责管理各种公共提示词，并调用 L1b 持久化服务存储到 `prompts.json`
4. **更新持久化文档**：在 L1b_持久化与IO基础设施.md 中增加 `prompts.json` 存储规范

### 6. Session 实体数据结构不一致

#### 设计文档要求（L1b_持久化与IO基础设施.md）
- Session 字段：`entity_id`, `project_id`, `name`, `status`, `messages`（内嵌消息对象数组）, `task_group_id`, `created_at`, `updated_at`
- **设计原则**：Message 不会脱离 Session 复用，应作为 Session 聚合的一部分内嵌存储

#### 实际代码实现（S01_data_persistence.py）
- Session 字段：`entity_id`, `project_id`, `messages`（数组，包含完整消息对象）, `status`, `summary`, `created_at`, `updated_at`

**不一致说明：**
- 设计要求使用 `messages` 数组内嵌存储消息对象（符合 DDD 聚合根原则）
- 代码实现与设计一致，但缺少 `name` 和 `task_group_id` 字段

**影响：**
- 缺少 `name` 字段：无法为会话命名
- 缺少 `task_group_id` 字段：无法关联任务组

**设计文档更新建议：**
1. **确认 Session 内嵌 messages 设计**：Message 作为 Session 聚合的一部分，不独立存储
   - **理由**：Message 不会脱离 Session 复用，符合 DDD 聚合根原则
   - **优势**：一次查询获取全部消息、数据天然一致、避免孤儿消息
2. **统一 Session 数据结构**：
   ```json
   {
     "entity_id": "sess-123",
     "project_id": "proj-456",
     "name": "会话名称",
     "status": "active",
     "messages": [
       {
         "message_id": "msg-001",
         "order_index": 0,
         "role": "user",
         "content": "用户问题",
         "created_at": "2026-05-18T10:00:00"
       },
       {
         "message_id": "msg-002",
         "order_index": 1,
         "role": "assistant",
         "content": "助手回答",
         "created_at": "2026-05-18T10:00:05"
       }
     ],
     "task_group_id": "tg-789",
     "created_at": "2026-05-18T10:00:00",
     "updated_at": "2026-05-18T10:30:00"
   }
   ```
3. **移除 messages.json**：不再需要独立的 `messages.json` 文件，消息随 Session 存储
4. **更新 L1b 文档**：在 L1b_持久化与IO基础设施.md 中明确 Session 内嵌 messages 的设计原则

### 7. 事件类型命名不一致

#### 设计文档要求（L1d_事件管理基础设施.md）
- `message.created`（消息创建）
- `llm.request_sent`（LLM请求发送）
- `llm.response_received`（LLM响应接收）

#### 实际代码实现（S03_event_plugin.py）
- `message.added`（消息添加）
- `llm.request.created`（LLM请求创建）

**不一致说明：**
- 事件类型命名不一致

**影响：**
- 事件订阅者可能无法正确订阅事件
- 事件追踪和审计困难

**设计文档更新建议：**
1. **统一事件类型命名规范**：采用 `resource.action` 格式（全部小写，句点分隔）
2. **标准化事件类型列表**：
   ```
   # 消息事件
   message.created      # 消息创建
   message.updated      # 消息更新
   message.deleted     # 消息删除

   # 会话事件
   session.created     # 会话创建
   session.updated     # 会话更新
   session.deleted     # 会话删除

   # 项目事件
   project.created     # 项目创建
   project.updated     # 项目更新
   project.deleted     # 项目删除

   # LLM 事件
   llm.request_sent    # LLM 请求发送
   llm.response_received  # LLM 响应接收
   llm.stream_chunk    # 流式分片
   llm.error           # LLM 错误

   # 工具事件
   tool.execution_started   # 工具执行开始
   tool.execution_completed # 工具执行完成
   tool.execution_failed    # 工具执行失败
   ```
3. **更新代码实现**：将 `MessageAdded` 改为 `message.created`，`llm.request.created` 改为 `llm.request_sent`
