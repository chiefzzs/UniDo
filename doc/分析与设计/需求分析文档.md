# 智能对话助手系统 - 需求分析文档

---

## 1. 需求背景

### 1.1 业务背景

本系统是一个基于大语言模型（LLM）的智能对话助手应用，旨在为用户提供自然语言交互能力，支持任务执行、代码开发、文档创作等多种场景。系统采用分层架构设计（L1-L5），具备良好的可扩展性和可维护性。

### 1.2 项目目标

- 提供流畅的对话体验
- 支持多场景任务执行（代码开发、创意写作、数据处理等）
- 实现完整的记忆和状态管理
- 提供可扩展的工具调用系统
- 支持多平台部署和多模型切换

## 3. 用户痛点问题

### 3.1 当前系统的问题

| 痛点编号 | 问题描述 | 影响范围 | 严重程度 |
|---------|---------|---------|---------|
| P001 | 工具调用失败时仅返回模拟成功消息，无法真正执行操作 | 所有工具调用场景 | 高 |
| P002 | ToolResult对象属性访问错误（使用`content`而非`result`） | WebSocket消息返回 | 高 |
| P003 | 工具查找失败后无法回退到配置驱动注册表 | 工具执行 | 中 |
| P004 | 工具执行结果格式不一致，难以统一处理 | 结果解析 | 中 |
| P005 | 缺少完整的工具执行验证测试 | 质量保障 | 中 |

### 3.2 用户体验痛点

- **工具调用不透明**：用户无法确认工具是否真正执行
- **错误反馈不及时**：工具执行失败时缺少明确的错误信息
- **执行结果不可追溯**：工具调用记录不完整，难以排查问题
- **跨平台兼容性差**：不同操作系统下工具行为不一致

## 4. 新工作流设计

### 4.1 工具执行工作流

```
用户请求 → 意图识别 → 工具选择 → 工具查找 → 参数验证 → 真实执行 → 结果返回 → 持久化
              ↓              ↓              ↓
         L3a意图服务    L2f工具管理    L2c工具执行
              ↓              ↓              ↓
         解析用户意图    获取工具定义    调用真实实现
```

### 4.2 工作流步骤说明

| 步骤 | 组件 | 职责 | 输入 | 输出 |
|-----|------|------|------|------|
| 1 | 用户界面 | 接收用户请求 | 用户输入文本 | 请求对象 |
| 2 | L3a意图服务 | 分析用户意图 | 用户输入 | 意图分析结果 |
| 3 | L3a任务协调 | 确定执行路径 | 意图结果 | 工具调用指令 |
| 4 | L2f工具管理 | 获取工具定义 | 工具名称/ID | 工具配置 |
| 5 | L2c工具执行 | 执行工具调用 | 工具参数 | 执行结果 |
| 6 | L1b持久化 | 存储调用记录 | 调用信息 | 存储确认 |
| 7 | L1d事件总线 | 发布执行事件 | 事件数据 | 事件确认 |
| 8 | 用户界面 | 展示执行结果 | 结果数据 | 可视化输出 |

## 5. 新旧工作流对比

### 5.1 工具执行流程对比

| 对比维度 | 旧工作流 | 新工作流 |
|---------|---------|---------|
| **工具查找** | 仅使用工具注册表 | 支持配置驱动注册表回退 |
| **执行方式** | 模拟执行（找不到工具时返回假成功） | 真实执行（找不到工具时返回错误） |
| **结果属性** | 使用`content`属性 | 使用`result`属性 |
| **错误处理** | 静默失败 | 明确错误信息 |
| **跨平台支持** | 有限 | 完整支持Windows/Linux/macOS |

### 5.2 数据流向对比

**旧工作流：**
```
用户请求 → L3 → L2c → （工具查找失败）→ 返回模拟成功
```

**新工作流：**
```
用户请求 → L3 → L2c → （工具查找失败）→ 尝试配置注册表 → 真实执行 → 返回真实结果
```

### 7.1 场景SC017：代码开发 - 工具执行

**触发界面**：对话界面输入框

**输出**：

| tool_result | ToolResult | 工具执行结果 |
| task_status | TaskStatus | 任务状态 |
| execution_history | list | 执行历史记录 |

**数据流向**：

```
1. 用户输入 → L5对话界面 → L4b WebSocket网关
2. WebSocket请求 → L3b场景服务 → 意图识别
3. 意图分析 → 确定工具调用 → L3a任务协调
4. 工具调用指令 → L2c工具执行服务
5. L2c查找工具 → L2f工具管理服务

7. 找到工具 → 执行真实工具实现
8. 执行结果 → 返回L3a → 更新任务状态
9. 任务状态 → 持久化到L1b
10. 执行事件 → 发布到L1d事件总线
11. 结果 → 返回L4b → 推送至L5界面
```

### 9.5 意图分析执行路径

| 执行路径 | 触发条件 | 处理组件 | 说明 |
|---------|---------|---------|------|
| **DIRECT_COMPLETION** | 简单问题，无需工具调用 | BaseExecutionService | 直接完成，返回文本响应 |
| **SINGLE_TOOL** | 需要调用单个工具 | ToolTaskExecutor → ToolGroupExecutor | 通过ToolGroupExecutor统一调度执行 |
| **TOOL_GROUP** | LLM返回多个工具调用 | ToolGroupExecutor | 管理和执行工具组，保留原始tool_call_id |
| **TASK_GROUP** | 复杂任务，需要多步骤执行 | TaskGroupExecutor → ToolGroupExecutor | 创建子任务组，通过ToolGroupExecutor执行工具调用 |

### 9.5.1 工具组执行流程说明

**ToolGroupExecutor作为统一工具执行入口**，支持单工具和多工具调用场景：

1. **入口统一**：无论是SINGLE_TOOL还是TOOL_GROUP路径，最终都通过ToolGroupExecutor执行
2. **工具调用解析**：DialogueBasedLLMService解析LLM响应中的tool_calls列表
3. **工具组创建**：ToolGroupExecutor接收工具调用列表，保留每个调用的原始ID（不修改）
4. **逐个执行**：遍历工具列表，依次调用L2c工具执行服务
5. **结果封装**：每个工具执行完成后，封装为`role: "tool"`消息格式，包含tool_call_id用于关联
6. **上下文累积**：工具结果追加到messages数组
7. **第二轮调用**：携带完整上下文再次调用LLM，生成总结回复

**共享执行流程优势**：
- 统一的工具执行入口，简化代码维护
- 一致的工具结果格式（role: "tool"消息）
- 统一的tool_call_id管理，支持追踪和关联
- 简化messages数组累积逻辑

### 9.6 工具组执行器（ToolGroupExecutor）

**职责**：管理和执行大模型返回的多个工具调用

**所属文件**：`src/services/L3_scenario_coordination/L3a_task_coordination/tool_group_executor.py`

**执行流程**：
1. **创建**：依据大模型返回的tool_calls信息创建工具组对象
   - 解析LLM返回的多个工具调用定义（支持function格式和简化格式）
   - 保留每个工具调用的原始ID（不修改，直接使用LLM生成的ID）
   - 发布工具组创建消息

2. **执行工具组**：
   - 遍历工具组中的每个工具调用
   - 利用大模型返回的信息，调用L2c工具执行服务的`execute_tool_with_message_format`方法
   - 每个工具执行完成后，发布工具执行完成事件
   - 将工具执行结果封装为role="tool"消息，包含tool_call_id用于关联

3. **完成**：所有工具执行完毕后，返回工具执行结果列表

**工具调用格式支持**：
```json
// 格式1：function格式（标准OpenAI格式）
{
  "id": "call_xxx",
  "function": {
    "name": "LS",
    "arguments": "{\"path\": \"/workspace\"}"
  }
}

// 格式2：简化格式
{
  "id": "call_xxx",
  "name": "LS",
  "arguments": "{\"path\": \"/workspace\"}"
}
```

### 9.7 流式响应处理机制

**StreamMerger组件**负责处理LLM流式响应的合并：

| 数据类型 | 流式输出字段 | 处理方式 | 输出格式 |
|---------|------------|---------|---------|
| 思考数据 | reasoning_content | 实时转发展示，不保存到messages | {"type": "thinking", "content": "..."} |
| 文本数据 | content | 累积合并，保存到messages | {"type": "content", "content": "..."} |
| 工具数据 | tool_calls | 累积合并arguments，保留原始ID | {"type": "content", "tool_calls": [...]} |

### 9.8 历史消息累积机制

**所属文件**：`src/services/L3_scenario_coordination/L3a_task_coordination/dialogue_based_llm_service.py`

**DialogueBasedLLMService**的`build_messages_from_history`方法负责构建完整的messages数组：

1. **系统消息**：始终作为第一条消息，包含工具信息
2. **用户消息**：累积所有用户输入，包含环境上下文信息
3. **助手消息**：累积所有助手响应，包含tool_calls字段（如果有）
4. **工具消息**：累积所有工具执行结果，包含tool_call_id字段用于关联

**消息格式示例**：
```json
{
  "role": "assistant",
  "content": "我将帮你创建目录...",
  "tool_calls": [
    {"id": "call_xxx", "name": "LS", "arguments": "{...}"}
  ]
}
```

```json
{
  "role": "tool",
  "tool_call_id": "call_xxx",
  "content": "<toolcall_status>done</toolcall_status>\n<toolcall_result>...</toolcall_result>"
}
```

**累积时机规范**：

| 阶段 | 触发时机 | 负责组件 | 添加内容 |
|------|---------|---------|---------|
| 阶段1 | LLM返回响应后 | DialogueBasedLLMService | assistant消息（含content、tool_calls） |
| 阶段2 | 工具执行完成后 | ToolGroupExecutor | tool消息（含tool_call_id、content） |
| 阶段3 | 用户输入后 | IntentService | user消息 |

## 11. 异常情况汇总

| 场景 | 异常类型 | 触发条件 | 处理方式 | 用户反馈 |
|-----|---------|---------|---------|---------|
| 工具执行 | 工具不存在 | 工具名称/ID未注册 | 返回错误 | "未找到工具：{tool_name}" |
| 工具执行 | 参数错误 | 缺少必需参数 | 返回错误 | "缺少必需参数：{param}" |
| 工具执行 | 执行失败 | 工具执行异常 | 返回错误 | "工具执行失败：{error}" |
| 工具执行 | 超时 | 执行时间超限 | 返回错误 | "工具执行超时" |
| 会话管理 | 会话不存在 | session_id无效 | 返回错误 | "会话不存在" |
| 项目管理 | 权限不足 | 用户无访问权限 | 返回错误 | "权限不足" |
| LLM调用 | API错误 | 大模型服务异常 | 重试/降级 | "服务暂时不可用，请稍后重试" |
