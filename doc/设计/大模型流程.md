# 大模型流程设计：工具调用前置任务机制

## 一、问题背景

当大模型返回响应时，可能同时包含：
1. **自然语言回复**：如"我来为您创建 test-2 目录。首先让我查看一下当前的目录结构..."
2. **工具调用**：如调用 `LS` 查看目录

当前实现中，自然语言回复可能被忽略或与工具调用结果混合返回，导致用户体验不够流畅。

## 二、设计目标

1. **提升用户体验**：在执行工具调用前，先告知用户操作意图
2. **流程清晰化**：明确区分"意图说明"和"实际执行"两个阶段
3. **可扩展性**：支持多种类型的前置任务
4. **可观察性**：通过事件总线监控整个流程

|------|------|------|

| **PreTaskQueue** | 管理前置任务 | 支持优先级排序和状态追踪 |
| **TaskOrchestrator** | 编排任务执行 | 协调前置任务和工具调用的顺序 |
| **EventBus** | 事件发布/订阅 | 提供流程可观察性 |

## 四、核心组件设计

### 4.1 ResponseParser（响应解析器）

**职责**：从LLM响应中提取自然语言内容和工具调用，判断是否需要前置任务。

**关键逻辑**：
- 检测前置提示词：`首先`、`先`、`第一步`、`先看`、`先检查`、`先确认`
- 判断响应类型：直接回答 / 需要工具调用

**输出结构**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `natural_language_content` | Optional[str] | 自然语言回复内容 |
| `tool_calls` | List[Dict] | 工具调用列表 |
| `is_direct_answer` | bool | 是否直接回答（无工具调用） |
| `pre_task_hint` | Optional[str] | 前置任务提示（检测到前置关键词时） |

### 4.2 PreTaskQueue（前置任务队列）

**职责**：管理需要在工具调用前执行的临时任务。

**前置任务类型**：

| 类型 | 说明 | 优先级 |
|------|------|--------|
| `INFORM_USER` | 告知用户操作意图 | 1（最高） |
| `VALIDATE_CONTEXT` | 验证上下文 | 2 |
| `PREPARE_ENV` | 准备环境 | 3 |
| `CONFIRM_ACTION` | 确认操作 | 4（最低） |

**任务结构**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `task_id` | str | 任务唯一标识 |
| `task_type` | PreTaskType | 任务类型 |
| `description` | str | 任务描述 |
| `priority` | int | 优先级（数字越小越优先） |
| `metadata` | Dict | 附加元数据 |
| `completed` | bool | 完成状态 |

### 4.3 TaskOrchestrator（任务编排器）

**职责**：协调前置任务和工具调用的执行顺序。

**执行流程**：
1. 解析LLM响应
2. 检测是否有前置任务提示
3. 创建前置任务（如有）
4. 按优先级执行前置任务
5. 发布工具执行开始事件
6. 执行工具调用

### 5.2 事件序列

| 序号 | 事件类型 | 触发时机 | 说明 |
|------|----------|----------|------|
| 1 | `llm.response_received` | LLM响应接收后 | 包含解析结果 |
| 2 | `pre_task.created` | 前置任务创建时 | 记录任务信息 |
| 3 | `pre_task.executing` | 前置任务执行前 | 通知开始执行 |
| 4 | `client.message_sent` | 前置任务消息发送 | 告知用户意图 |
| 5 | `pre_task.completed` | 前置任务完成后 | 标记任务完成 |
| 6 | `tool.execution_started` | 工具执行开始前 | 发布工具调用列表 |
| 7 | `tool.call_started` | 单个工具调用开始 | 记录工具调用 |
| 8 | `tool.call_completed` | 单个工具调用完成 | 记录执行结果 |
| 9 | `tool.execution_completed` | 所有工具执行完成 | 汇总执行结果 |
| 10 | `dialog.completed` | 对话完成 | 结束对话 |

## 十一、事件持久化开关机制

### 11.1 设计目标

- **灵活性**：支持动态控制事件持久化行为
- **性能优化**：可选择只存储关键信息，减少存储开销
- **调试支持**：缺省打开详细信息持久化，方便问题定位

### 11.2 配置参数

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `persistence_enabled` | bool | True | 是否启用事件持久化 |
| `persist_full_payload` | bool | True | 是否持久化完整的payload |
| `memory_cache_enabled` | bool | True | 是否启用内存缓存 |

### 11.3 使用方式

```python
# 获取事件总线配置
event_bus = get_event_bus()
config = event_bus.get_config()

# 关闭持久化
config.persistence_enabled = False

# 只存储关键信息（减少存储开销）
config.persist_full_payload = False
```

### 11.4 事件发布接口扩展

```python
# 发布事件时可指定是否持久化
event_bus.publish(event, correlation_id=None, persist=None)
# persist=None: 使用全局配置
# persist=True: 强制持久化
# persist=False: 强制不持久化
```

## 十二、工具执行实时输出机制

### 12.1 设计目标

- **实时性**：工具执行输出实时推送到前端
- **流式传输**：类似LLM流式输出，分块推送
- **可识别性**：前端可识别并实时更新到对应位置

### 12.2 事件类型

| 事件类型 | 说明 | 触发时机 |
|----------|------|----------|
| `tool.execution_output` | 工具执行输出块 | 每次接收到输出时 |
| `tool.execution_output_end` | 工具输出结束 | 工具执行完成时 |

### 12.3 事件结构

**tool.execution_output**：

```json
{
    "event_type": "tool.execution_output",
    "payload": {
        "call_id": "call-abc123",
        "tool_name": "RunCommand",
        "output": "命令执行输出内容...",
        "source_component": "L2_tool_execution",
        "source_service": "ToolExecutor"
    },
    "timestamp": "2026-05-20T10:00:00.000000"
}
```

**tool.execution_output_end**：

```json
{
    "event_type": "tool.execution_output_end",
    "payload": {
        "call_id": "call-abc123",
        "tool_name": "RunCommand",
        "source_component": "L2_tool_execution",
        "source_service": "ToolExecutor"
    },
    "timestamp": "2026-05-20T10:00:05.000000"
}
```

### 12.4 实现机制

```
┌──────────────────────────────────────────────────────────────────┐
│                    工具执行实时输出流程                          │
├──────────────────────────────────────────────────────────────────┤
│                                                                │
│  ToolExecutor ──► output_callback(chunk) ──► _on_tool_output   │
│       │                                            │           │
│       │                                            ▼           │
│       │                                   EventBus.publish     │
│       │                                   (tool.execution_output)│
│       │                                            │           │
│       │                                            ▼           │
│       │                                   WebSocketServer       │
│       │                                   _on_event_received   │
│       │                                            │           │
│       │                                            ▼           │
│       │                                   _broadcast_event     │
│       │                                            │           │
│       │                                            ▼           │
│       └──────────────────────────────────────► Client UI        │
│                                                  │             │
│                                                  ▼             │
│                                           更新输出区域         │
│                                                                │
└──────────────────────────────────────────────────────────────────┘
```

### 12.5 前端处理

前端收到 `tool.execution_output` 事件时：

1. 根据 `call_id` 找到对应的工具调用
2. 将 `output` 追加到该工具调用的输出区域
3. 自动滚动到底部显示最新内容

前端收到 `tool.execution_output_end` 事件时：

1. 标记该工具调用的输出完成
2. 可以显示"执行完成"状态

## 十四、总结

本设计通过引入 **ResponseParser**、**PreTaskQueue** 和 **TaskOrchestrator** 三个核心组件，实现了工具调用前的前置任务机制。主要优势：

1. **用户体验提升**：执行工具前先告知用户操作意图
2. **流程清晰化**：明确区分意图说明和实际执行
3. **可扩展性强**：支持多种前置任务类型
4. **可观察性好**：通过事件总线监控整个流程
5. **灵活性高**：支持事件持久化开关配置
6. **实时性强**：工具执行输出实时推送到前端

该设计可以无缝集成到现有架构中，只需在 WebSocket 消息处理器中添加对 `TaskOrchestrator` 的调用即可。
