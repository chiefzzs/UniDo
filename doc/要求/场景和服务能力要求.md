1、文档接口：
    1） 用户场景分析
    2） 基于场景提取分层职责
    3） 基于每个场景，进行场景分析
        + 每个场景需要描述控制流： 描述在层内、层间分析
        + 每个场景需要描述数据流： 描述在层内、层间分析
  
2、进行场景分析：
    + 照用户时间轴操作次序/依赖次序来描述场景
    + 每个场景，所需的服务、能力，按照系统处理的信息的时间轴、数据流加工的先后次序来进行
    + 确保用户操作时间轴中不要有缺项
    + 确保每个场景系统中各服务处理信息流不要无依据， 不要次序颠倒，不要缺项
    + 述方式： xxx场景：  xxx用户输入什么数据； xxx层xxx服务处理xxx数据 ； xxx层xxx服务输出xxx数据 ； xxx用户继续操作xxx数据
  
3、每个服务：
    + 必须是独立的，所有服务加起来是全面的
    + 服务需要分层或者分区管理
    + 每个服务务必须描述职责、数据、接口
    + 核心数据必须采用类图描述
    + 每个职责必须描述操控的数据
    + 每个接口必须有功能描述
    + 必须有一张表格： 描述核心职责，采用什么接口，和什么数据来支持的。（必须按数据流、控制流的时间顺序描述）
    + 每个服务必须描述对其他服务的依赖： 必须清除的描述是什么数据依赖
    + 核心类图中表达清除服务间的数据依赖

4.类图关系必须遵从领域模型规则
    1、聚合： 上级对象必须有下级对象的属性
    2、组合： 上级对象必须有下级对象的属性
    3、关联： 上级对象必须有下级对象的属性
    5、继承： 下级对象必须是上级对象
    6、实现： 上级对象必须有下级对象的属性

5、核心流程抽象要求：
    1） 流程(L3 层） 与原子服务分离; 控制与实现分离
    2） 业务流（L3层）要由足够抽象： 一个业务流（L3层）可支持多个场景(UI 层)
    3） 子服务（L2层），实现中基于业务原子概念描述 ，和具体的通用的实现(L1层)分离
    4） 多个业务原子服务（L2层），使用一个通用的原子服务（L1层）

6、支持可测试、可定位
    1) 所有的工具内部交互可定位： 层/组件间切换需要发布事件，事件需要持久化保存，方便测试检测流程是否正确
    2) 所有与外部系统交互可定位： 
        + 和外部系统的交互过程数据和序列需要按照保存： 比如和大模型的交互，支持测试分析和定位
        + 和外部熊交互过程如果外部有持久化的保存的，可以不用保存。 
    3） 上述可定位机制是一套，不能单独使用
