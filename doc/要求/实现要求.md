1、采用服务化方式实现。放到src/services/xxx层/xxx子层（可选）/xxx服务.py
2、严格按照职责来开发，优先定义外部接口，依据外部接口和概念模型来实现
3、非必要不能修改外部接口和概念模型。
4、每个服务内部必须描述数据处理流程：
    1）接收的数据结构
    2）内部处理步骤
    3）输出的数据结构
    4）调用的下层服务

5、核心流程实现要求：
    1）流程a)输入选择工具执行流程：无状态，消息管理由调用方负责，调用方在调用前先追加输入到历史
    2）流程b)任务组调度流程：负责管理Task循环，每个Task调用流程a执行，工具执行结果由流程b追加到历史
    3）流程c)用户对话流程：用户对话入口，协调流程a执行用户请求，用户消息由流程c追加到历史

6、数据流流转要求：
    1）L4层 → L3层 → L2层 → L1层
    2）关键数据流：用户输入 → 会话历史 → LLM请求 → LLM响应 → 工具调用 → 工具结果 → 最终响应
    3）组件间数据流转必须清晰描述，每个环节的输入输出必须明确

7、服务职责边界要求：
    1）流程a)涉及组件：S07/S11/S19/S21/S17/S20/S18/S06/S01
    2）流程b)涉及组件：S12/S13/S06/S01 + 流程a)所有组件
    3）流程c)涉及组件：S14/S15/S16/S10/S06 + 流程a)/流程b)所有组件

8、关联数据设计原则：
    1）项目关联配置（工作区、模型等）应只保存关联ID（如workspace_id、model_config_id），而非完整配置对象
    2）配置详情应在需要时动态加载，避免数据冗余和同步问题
    3）依赖数据库生成ID的后续写操作（如关联表写入），ID必须来自持久化后的实体对象，禁止使用请求入参中的ID
    4）前端表单应只提供选择已有配置的方式，禁止同时提供手动输入配置详情的功能，避免数据冲突

9、业务流程顺序原则：
    可复用对象关联原则：使用实体ID关联数据，避免直接操作对象实例
    1）create模板：组装实体 → save实体（获得ID）→ 使用ID写关联数据 → 再写其他聚合数据
    2）update模板：读取原实体 → 合并更新字段 → save实体（确保ID稳定）→ 使用实体ID进行关联数据的增删改 → 再写其他数据
    3）在扩展字段时，严禁顺带重排与该字段无关的关键链路；若必须调整顺序，需先抽象出明确的业务不变量

10、关联数据写入约束：
    1）写关联数据前必须保证关联ID已从持久化实体取得
    2）关联数据写入逻辑应封装为独立步骤/函数，函数入参仅接受已确定的关联ID
    3）禁止在关联数据写入步骤内回退使用请求中的ID字段
