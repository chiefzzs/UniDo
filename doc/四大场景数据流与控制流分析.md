
# 四大场景数据流与控制流分析（基于新分层架构）

**项目**: Trae IDE AI助手 + LLM代理系统  
**基于设计**: 分层架构设计建议（MECE原则）  
**日期**: 2026-05-18

---

## 一、场景概览

### 1.1 四个核心场景定义

| 场景编号 | 场景名称 | 复杂度 | 核心特征 |
|---------|---------|--------|---------|
| **SC04** | 简单单次文本对话 | 简单 | 无工具调用，直接LLM返回文本 |
| **SC05** | 简单单次工具对话 | 中等 | LLM选择并调用单个工具，返回结果 |
| **SC16** | 任务组对话 | 复杂 | 拆解为多个任务，顺序/并行执行 |
| **SC17** | 嵌套任务组对话 | 非常复杂 | 任务组中嵌套子任务组，递归执行 |

### 1.2 分层对照

| 原分层 | 新分层 | 变化说明 |
|-------|--------|---------|
| UI应用层 (L5) | L5 UI应用层 | 保留 |
| 网关层 (L4) | L4 网关层 | 保留 |
| 对话层 (L3) | L3 会话与协调层 | 简化，移除意图理解 |
| 领域层 (L2) | L2 任务执行与领域层 | 强化，整合意图理解 |
| 基础设施层 (L1) | L1 基础设施层 | 拆分为子层，增加 L1.5 记忆与状态层 |

## 二、SC04: 简单单次文本对话

### 2.1 场景描述
用户输入纯文本问题，LLM直接返回文本回答，不涉及工具调用。

**典型示例**: 用户问"你好"、"介绍一下你自己"、"今天天气怎么样"等无需工具的问题。

### 2.2 核心数据流程

#### 2.2.3 数据流变化过程

| 阶段 | 数据形态 | 数据来源 | 数据去向 | 关键变化 |
|-----|---------|---------|---------|---------|
| 用户输入 | UserMessage对象 | L5 UI | L4 网关 | 封装为结构化消息 |
| 会话验证 | Session对象 | L3 会话层 | L2 领域层 | 验证会话有效性 |
| 消息存储 | Message实体 | L2b 领域层 | L1.5 记忆层 | 持久化用户消息 |
| 意图分析 | IntentResult | L2a 任务层 | L2a 任务层 | 决策为DIRECT分支 |
| LLM调用 | LLMRequest | L2c 工具层 | L1c LLM | 构建请求并调用 |
| 流式响应 | StreamChunk | L1c LLM | L4 网关 | 逐块接收响应 |
| 消息存储 | AssistantMessage | L2b 领域层 | L1.5 记忆层 | 持久化助手回复 |
| 结果渲染 | RenderedOutput | L4 网关 | L5 UI | 渲染最终响应 |

### 2.3 控制流与事件序列

| 序号 | 事件/操作 | 发布层 | 接收层 | 说明 |
|-----|---------|-------|-------|------|
| 1 | UserInputReceived | L5 UI应用层 | L4 网关层 | 用户输入消息 |
| 2 | RequestValidated | L4 网关层 | L3 会话层 | 身份验证通过 |
| 3 | SessionCreated/Resumed | L3 会话层 | L2 领域层 | 会话创建/恢复 |
| 4 | MessageAdded | L2b 领域层 | L1.5 记忆层 | 用户消息添加 |
| 5 | IntentAnalysisStarted | L2a 任务执行层 | L3 会话层 | 意图理解开始 |
| 6 | TextChunkReceived | L2c 工具访问层 | L4 网关层 | 流式文本块到达 |
| 7 | StreamOutput | L4 网关层 | L5 UI应用层 | 流式输出推送 |
| 8 | TaskCompleted | L2a 任务执行层 | L2b 领域层 | 任务完成 |
| 9 | MessageAdded | L2b 领域层 | L1.5 记忆层 | 助手消息添加 |
| 10 | ConversationCompleted | L3 会话层 | L5 UI应用层 | 对话完成 |

## 三、SC05: 简单单次工具对话

### 3.1 场景描述
用户输入需要工具调用的请求（如"读取README文件"），LLM选择并调用单个工具，工具执行后返回结果，LLM最终聚合结果返回文本回答。

**典型示例**: 用户问"帮我读取项目的README.md文件"、"搜索代码库中的配置类"等需要工具执行的问题。

### 3.2 核心数据流程

#### 3.2.1 工具调用请求结构（参考llm_proxy_records.json）

```json
{
  "timestamp": "2026-05-12 12:38:21",
  "user_text": "帮我读取README.md文件",
  "request": {
    "model": "Qwen/Qwen3.5-397B-A17B",
    "max_completion_tokens": 16000,
    "stream": true,
    "tools": [
      {
        "type": "function",
        "function": {
          "name": "Read",
          "description": "Reads a file from the local filesystem...",
          "parameters": {
            "type": "object",
            "properties": {
              "file_path": {
                "type": "string",
                "description": "The absolute path to the file to read."
              },
              "offset": {"type": "integer"},
              "limit": {"type": "integer", "minimum": 1}
            },
            "required": ["file_path", "limit"]
          }
        }
      }
    ],
    "messages": [
      {"role": "system", "content": "你是一个AI助手，可以调用工具..."},
      {"role": "user", "content": "帮我读取README.md文件"}
```

## 四、SC16: 任务组对话

### 4.1 场景描述
用户输入复杂任务请求（如"分析项目结构，生成README，更新package.json"），LLM拆解为多个任务，组成任务组，顺序或并行执行所有任务，最终聚合结果返回。

**典型示例**: 用户要求"分析我的Python项目，检查依赖安全性，然后生成API文档"。

### 4.2 任务组数据结构

#### 4.2.1 任务组定义结构

```json
{
  "task_group_id": "group-abc123",
  "name": "项目分析与文档生成",

  "status": "running",
  "execution_mode": "sequential",
  "tasks": [
    {
      "task_id": "task-001",
      "name": "分析项目结构",
      "description": "读取项目目录结构和关键文件",
      "status": "completed",
      "type": "single_tool",
      "tool_info": {
        "tool_name": "LS",
        "params": {"path": "/workspace/project"}

    },
    {
      "task_id": "task-002",
      "name": "检查依赖安全性",
      "description": "扫描项目依赖的安全漏洞",
      "status": "running",
```

## 五、SC17: 嵌套任务组对话

### 5.1 场景描述
用户输入超复杂任务请求，LLM拆解为任务组，其中部分子任务本身也是任务组（嵌套结构），递归执行所有层级，最终聚合结果返回。

**典型示例**: 用户要求"分析我的微服务架构项目，包括：1)分析每个服务的代码结构；2)检查服务间依赖关系；3)生成架构文档和部署建议"。

| L4 网关层 | 从 L5 接收请求、从下层接收事件 | 转发请求、发布事件 | API响应、WebSocket推送 |

| L2 任务执行与领域层 | 从 L1.5 读取记忆、从 L1 读取配置 | 更新 L2b 领域实体、写入 L1.5 记忆 | 任务结果、领域事件 |
| L1.5 记忆与状态层 | 从 L2 接收数据 | 存储消息、会话状态 | 历史记录、状态快照 |
| L1 基础设施层 | 从持久化存储读取 | 写入持久化存储 | 配置数据、持久化实体 |

## 八、四大场景数据流对比总结

### 8.1 数据流复杂度对比

| 场景 | 复杂度 | 工具调用 | 任务组 | 嵌套深度 | 关键数据流特征 |
|-----|--------|---------|-------|---------|--------------|
| **SC04** | 简单 | 无 | 无 | 1层 | 用户输入 → LLM → 直接回复 |
| **SC05** | 中等 | 单工具 | 无 | 1层 | 用户输入 → LLM → 工具执行 → LLM → 回复 |
| **SC16** | 复杂 | 多工具 | 单层 | 1-2层 | 用户输入 → LLM → 任务组 → 多任务执行 → LLM → 回复 |
| **SC17** | 非常复杂 | 多工具 | 多层嵌套 | N层 | 用户输入 → LLM → 嵌套任务组 → 递归执行 → LLM → 回复 |

### 8.2 数据流变化关键点

| 关键点 | SC04 | SC05 | SC16 | SC17 |
|-------|------|------|------|------|
| **意图分析决策** | DIRECT | SINGLE_TOOL | TASK_GROUP | TASK_GROUP |
| **工具执行** | ❌ | ✅（单次） | ✅（多次） | ✅（递归） |
| **任务检查/调整** | ❌ | ✅ | ✅ | ✅ |
| **任务组管理** | ❌ | ❌ | ✅ | ✅（嵌套） |
| **结果聚合** | 直接回复 | 工具结果聚合 | 多任务结果聚合 | 嵌套结果聚合 |
| **递归深度** | 1 | 1 | 2 | N |

### 8.3 统一数据流模式

所有场景共享的核心数据流模式：

```
用户输入 → 会话验证 → 消息持久化 → 意图分析 → 执行决策 → 
    [DIRECT → LLM直接回复]
    [SINGLE_TOOL → 工具执行 → 结果检查 → 调整 → LLM聚合]
    [TASK_GROUP → 任务组创建 → 任务循环执行 → 结果聚合]
 → 结果持久化 → UI渲染
```
