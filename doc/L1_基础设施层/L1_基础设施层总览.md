# L1 基础设施层 - 层总览

## 1. 层定位与职责

**L1 基础设施层** 是系统的最底层，提供核心的基础设施能力，不包含任何业务逻辑。**持久化是本层的核心基石**，所有组件的数据都通过持久化服务进行管理。

### 1.1 核心职责
- **持久化管理（L1b）**：作为底层核心，统一管理配置、LLM调用过程、事件等所有数据的持久化
- 管理系统配置（L1a）
- 封装LLM API调用（L1c）
- 实现事件发布-订阅（L1d）

### 1.2 设计原则
- **纯基础无业务**：不包含任何业务逻辑
- **可替换性**：各组件可独立替换
- **高可用性**：提供可靠的基础设施服务
- **性能优化**：针对高频操作进行优化
- **环境隔离**：通过环境变量实现开发与测试环境分离
- **数据完整性**：关键数据序列有序存储，保证数据一致性
- **可审计性**：采用完美JSON格式存储，方便人工核查

## 7. 关键设计决策

### 7.0 架构设计原则（核心）

#### 核心原则：L1b 不应定义业务实体的具体数据结构

**职责边界：**

| 层级 | 职责 | 不应包含 |
|-----|------|---------|
| **L1b 持久化基础设施** | 提供通用的 JSON 文件读写能力、环境隔离、文件系统操作 | 业务实体的具体数据结构定义 |
| **L2 层领域服务** | 定义业务实体结构、数据验证、业务规则 | 文件系统操作细节 |
| **调用方式** | L2 将实体序列化为 dict → 调用 L1b.save() → L1b 写入文件 | 绕过 L1b 直接操作文件 |

**L1b 核心接口（通用、无业务耦合）：**

| 方法 | 功能 | 参数 | 返回值 |
|-----|------|------|-------|
| `save(entity_type, data)` | 保存实体 | `entity_type: str`, `data: dict` | `entity_id: str` |
| `load(entity_type, entity_id)` | 加载实体 | `entity_type: str`, `entity_id: str` | `Optional[dict]` |
| `list(entity_type, filters=None)` | 列出实体 | `entity_type: str`, `filters: dict` | `List[dict]` |
| `delete(entity_type, entity_id)` | 删除实体 | `entity_type: str`, `entity_id: str` | `bool` |

**好处：**

- **解耦**：L1b 可以独立演进，不依赖业务变化
- **复用**：L1b 可以服务于不同的业务模块
- **清晰**：职责边界明确，符合分层架构原则

### 7.1 组件独立性

L1 层的四个组件设计为相对独立，但都依赖L1b持久化组件：
- L1a 配置管理：独立管理配置，配置数据通过L1b持久化
- L1b 持久化：底层核心，独立实现存储，可替换存储后端
- L1c LLM基础设施：独立封装API，调用过程通过L1b持久化
- L1d 事件系统：独立实现事件总线，事件数据通过L1b持久化

### 7.2 事件驱动设计

所有组件都通过事件系统进行通信：
- 配置变更发布事件
- 数据变更发布事件
- LLM调用发布事件
- 上层组件订阅感兴趣的事件

### 7.3 配置优先

系统行为由配置驱动：
- 模型配置决定LLM调用参数
- 工作区配置决定文件操作路径
- 工具配置决定可用工具列表

### 7.4 持久化核心设计

**7.4.1 环境隔离机制**
- 通过启动时环境变量 `STORAGE_ENV` 隔离开发与测试环境
- `STORAGE_ENV=dev`：开发环境，存储到 `./data/dev/` 目录
- `STORAGE_ENV=test`：测试环境，存储到 `./data/test/` 目录
- 未设置时默认使用开发环境

**7.4.2 文件存储结构**
- 每种类型的信息使用一个独立的JSON文件
- 文件命名规范：`{entity_type}.json`（如 `projects.json`、`sessions.json`）
- 多个数据采用数组存储，例如：
  ```json
  [
    {"entity_id": "proj-123", "name": "Project 1"},
    {"entity_id": "proj-456", "name": "Project 2"}
  ]
  ```

**7.4.3 序列支持**
- 关键数据序列（如消息序列、事件序列）保持有序存储
- 序列通过 `order_index` 字段或数组顺序维护
- 支持序列的追加、插入、删除操作

**7.4.4 完美JSON格式**
- 所有JSON文件采用格式化输出（4空格缩进）
- 键名使用驼峰命名法
- 日期时间采用ISO 8601格式（如 `"2026-05-18T10:00:00"`）
- 便于人工核查和编辑

## 8. 容错与恢复

### 8.1 组件级容错

| 组件 | 容错策略 |
|-----|---------|
| L1a | 默认配置保障、配置验证、配置回滚 |
| L1b | 文件操作容错、原子写入、备份策略 |
| L1c | 错误处理、超时处理、模拟模式降级 |
| L1d | 异步持久化、故障恢复、数据清理 |

### 8.2 跨组件恢复

当某个组件故障时：
- L1b 故障：L1a 可使用内存缓存的配置
- L1c 故障：使用模拟模式继续运行
- L1d 故障：事件可能丢失，但业务流程继续
